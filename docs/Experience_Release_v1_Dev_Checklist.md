# 《倒水乐乐乐》首次微信体验版 — 研发自检与打包说明

> **文档用途**：供研发在提交**第一次上线体验版**前完成自检、构建产物确认与公众平台侧配置核对。  
> **建议代码基线**：Git 标签 **`v0.3.0`**（提交 `94cf4d1` 及之后若另有修复，请以实际打包提交为准）。  
> **引擎**：Cocos Creator **3.8.8**  
> **更新日期**：2026-05-01  

---

## 1. 体验版范围说明（给研发与对外文案对齐）

### 1.1 本版应稳定交付的能力

| 模块 | 说明 |
|------|------|
| 核心玩法 | 选关、倒水、通关结算、暂停、设置（与 v0.2 一致的主链路） |
| 微信登录 | `wx.login` 获取 code；失败有提示，不阻塞本地试玩 |
| 云开发初始化 | `wx.cloud.init` 启动期调用成功，无「please call wx.cloud.init first」 |
| 云存档（当前实现） | `getOpenId` 云函数获取用户标识；客户端 **`wx.cloud.database()`** 读写集合 **`kv_saves`**（按业务 key 查询，依赖集合创建者权限隔离） |
| 分享 | 主菜单「分享求助」、结算「分享炫耀」；主菜单侧含**每日首次**金币奖励与防刷逻辑 |
| 其他 | GM、签到、商店/挑战等入口若为占位或弱依赖，以**不崩溃、无误导严重文案**为准 |

### 1.2 本版已知限制（须在体验说明 / 版本介绍中写明）

- **好友排行榜**：当前为**占位 / mock**，**未接入开放数据域**，勿对外承诺真实好友榜。
- **分享回调**：微信分享成功判定以当前实现为准；若体验版仅验证流程，建议在说明中写「奖励以游戏内提示为准」。
- **分享与每日首次奖励**：若用户**当日先完成结算页「分享炫耀」再点主菜单「分享求助」**，存在与 `dailyShareCount` 耦合的边角情况，测试与策划需知晓（后续可单独字段拆解）。

## 2. 研发功能自检清单（打包前逐项勾选）

建议在 **微信开发者工具真机预览** + **至少一台真机**各跑一遍；开发者工具用于看 Console，真机用于验证存储与分享。

### 2.1 启动与基础

- [ ] 冷启动进入主菜单无红错崩溃。
- [ ] Console 可见 **`[WXAPI] wx.cloud.init ok`**（微信环境）。
- [ ] 主菜单版本展示与配置一致：`GAME_CONFIG.VERSION` 为 **`0.3.0`**（设置面板内版本号）。
- [ ] （可选）将主菜单底部文案由「测试版」调整为「**体验版**」，避免与对外称呼不一致（当前仓库约为主菜单 `v0.3 微信测试版 (cloudlog)`，可按发布需要改掉 debug 后缀）。

### 2.2 登录与会话

- [ ] 启动后触发登录链路，`[SDKManager] login code:` 日志存在且非空（失败场景下有 Toast / 错误写入存档字段）。
- [ ] 登录失败后可通过设置内「同步 / 重试」类入口再次触发（若有），且无死循环调用。

### 2.3 云存档

- [ ] **首次无云档**：提示或日志表明用本地初始化云端（与产品文案一致即可）。
- [ ] **通关或改设置后**：`kv_saves` 中对应用户记录 **`updatedAt` / `value`** 有更新。
- [ ] **杀进程重启**：进度与 GM 摘要一致，无异常回滚。
- [ ] **仅清本地存储**（模拟换端）：重启后能从云端恢复到最新进度（依赖库权限与网络）。
- [ ] **云端删档**：删除当前 `_openid` 的 `kv_saves` 记录后，重启应以全新云档开始。
- [ ] **多账号隔离**：3 个真实微信账号分别进入后，`kv_saves` 至少出现 3 条不同记录；`key` 形如 `water_sort_save_v1_cloud_<openid>`，且 `ownerOpenId` 分别对应不同账号。
- [ ] **旧共享档防串档**：新账号无匹配 `ownerOpenId` 时必须创建新云档，不得读取旧的共享 `water_sort_save_v1_cloud` 记录。

### 2.4 分享

- [ ] 主菜单「分享求助」可拉起分享；返回后 Toast 与 **`ShareManager`** 返回文案一致。
- [ ] **同一自然日第二次**点击「分享求助」：**不应重复发放**首次金币，应出现限制类提示（与当前实现一致）。
- [ ] 结算「分享炫耀」不崩溃，文案含关卡/步数/星等变量（若已实现）。

### 2.5 玩法与 v0.2 回归（最短路径）

- [ ] 进入关卡 → 倒水 → 通关 → 「下一关」。
- [ ] 暂停、设置开关、音效振动相关不崩溃。
- [ ] GM：摘要可读；重置后云侧与预期一致（注意重置会清空进度，测试账号慎用）。

### 2.6 排行榜入口（预期管理）

- [ ] 点击「排行」：仅验证**不崩溃**、可返回主菜单；列表为占位/mock **属预期**。

## 3. 微信侧环境核对（研发 / 运维）

### 3.1 小程序后台

- [ ] 小程序类目、名称、简介与体验版一致；无违禁素材。
- [ ] **开发 → 开发管理 → 体验版**：上传体验版后，体验成员已添加。
- [ ] 若使用云开发：控制台内 **云环境** 与开发者工具选用环境一致（默认环境 / envId）。

### 3.2 云数据库

- [ ] 集合 **`kv_saves`** 已创建。
- [ ] **权限与安全规则**：`kv_saves` 必须设置为仅创建者可读写/仅创建者可写读自身数据，**禁止**误配为全员可读写；这是 V0.5.3 存档隔离的上线阻断项。
- [ ] **存档记录结构**：新记录必须同时包含用户级 `key`、`ownerOpenId`、`value`、`updatedAt`；禁止多个账号持续更新同一条 `water_sort_save_v1_cloud` 旧记录。
- [ ] （若使用）`rank_data` 等与排行相关的集合：若体验版未使用真实排行，可暂不对外说明或保持最小权限。

### 3.3 构建目录中的云函数

- V0.5.3 必须保留并部署 `cloudfunctions/getOpenId`。每次 Cocos 重新构建后，确认 `build/wechatgame/project.config.json` 含 `cloudfunctionRoot`，且 `build/wechatgame/cloudfunctions/getOpenId` 存在。
- **上线前固定提醒**：Cocos Creator 的微信小游戏构建会覆盖 `build/wechatgame`，可能清掉云函数目录与 `cloudfunctionRoot`。每次 build 完必须重新确认/补回：
  - `build/wechatgame/cloudfunctions/getOpenId/index.js`
  - `build/wechatgame/cloudfunctions/getOpenId/package.json`
  - `build/wechatgame/project.config.json` 中的 `"cloudfunctionRoot": "cloudfunctions/"`
- 在微信开发者工具中看到 `cloudfunctions/getOpenId` 后，右键该云函数执行上传/部署依赖；控制台已有调用成功，只能说明当前云环境存在函数，不代表新构建产物里仍保留云函数目录。

## 4. Cocos 打包与产物检查

### 4.1 构建步骤（概要）

1. Creator 菜单：**项目 → 构建发布**。
2. 发布平台：**微信小游戏**。
3. 填写 AppID、路径等；执行构建。
4. 构建输出目录一般为项目下 **`build/wechatgame`**（以工程实际配置为准）。

### 4.2 产物自检

- [ ] 用 **微信开发者工具** 打开构建目录，能编译运行。
- [ ] 左侧资源管理器能看到 `cloudfunctions/getOpenId`；若看不到，先补回构建目录和 `cloudfunctionRoot` 再上传。
- [ ] `project.config.json` 中 **AppID**、**基础库**要求与后台一致。
- [ ] 首包体积与分包策略符合平台提示（关注单文件过大警告是否可接受）。
- [ ] 记录本次构建的 **Git commit id**，写入体验版发版说明或内部台账。
